## Generation Process

Each sprint runbook was generated by a subagent:
1. Loaded `command-runway-pattern` skill + templates
2. Read SPRINTS.md for that sprint's tasks, acceptance criteria, dependencies
3. Read relevant VAP/VAE spec sections
4. Generated PLAN.md (Layer 2) with stages mapped 1:1 to sprint tasks
5. Generated RUNBOOK.md (Layer 3) with same stages, empty execution log, goal verification, JSON extension

## Key Patterns Observed

### Sprint-to-Stage Mapping
- 1 sprint ≈ 1–3 stages (most sprints → 1 stage per task, some consolidated)
- Typical sprint: 5–12 stages, 40–80 commands total
- Backend-first ordering naturally emerged (pipeline → models → serving → UI)

### Discipline Enforced Across All 20 Runbooks
- **⏾→✎→✓** role tagging on every command
- **DAG via `depends_on`** — no stage advances until its ✓ commands pass
- **Relaxed inspect rule** — re-inspection after mutation allowed via DAG (e.g., generate → read → patch)
- **No gate weakening** — fallback actions never relax assertions
- **Retry tracking** — `Retry#` column in execution log
- **Machine-readable JSON** — `depends_on`, `expected` assertions, `content_ref` resolution

### VAE-Specific Adaptations
- **Conformance as first-class gate** — `tests/conformance/` stages in most sprints
- **Determinism requirements** — ML stages include `tests/conformance/ml-determinism/` with fixed-seed assertions
- **Load/chaos testing** — dedicated stages in hardening sprints (7, 11, 20)
- **Security/privacy as sprints, not afterthoughts** — Sprints 14, 15
- **UI deferred to Phase 3** — Browser SDK/extension only in Sprints 16–18
