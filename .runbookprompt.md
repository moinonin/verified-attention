# COMMAND_RUNWAY Plan + Runbook Generation Prompt

Use this prompt to generate a complete COMMAND_RUNWAY execution plan and runbook for a feature. The output is a `PLAN.md` (static plan with command tables) and `RUNBOOK.md` (execution template with empty log) following the three-layer taxonomy defined in the `command-runway-pattern` skill.

---

## Instructions

Given a feature spec, sprint goal, or task description, generate two files:

### 1. PLAN.md
A static plan with stages, commands, preconditions, verification gates, and success metrics. Uses the `stage-template.md` format.

### 2. RUNBOOK.md
An execution template ready for a coding agent to follow. Uses the `runbook-template.md` format with:
- Taxonomy (Feature → Stage → Command)
- Intent & Goals (Global + Local)
- Preconditions table
- Command Runway with ⏾/✎/✓ markers and `depends_on` chains
- Execution Log table (empty, to be filled during execution)
- Goal Verification (Local + Global checks)
- Machine-readable JSON extension at the end

---

## Mandatory Rules

### Python Virtual Environment Rule (CRITICAL)
**All Python backend commands MUST use the project's `.venv` virtual environment.**

1. **Precondition check**: Every runbook MUST include a precondition that `.venv` exists:
   - Check: `test -d .venv && .venv/bin/python --version`
   - If missing: the FIRST command in Stage A MUST be `python3 -m venv .venv` (or a specific Python version if required: `python3.11 -m venv .venv`)

2. **All Python invocations**: Every Python command in the runbook MUST use `.venv/bin/python` or `.venv/bin/pip` — NEVER bare `python` or `python3` or `pip`.

3. **Activation**: If a script sources the venv (`. .venv/bin/activate`), the activation MUST happen in the same shell invocation that runs the Python code. Do NOT assume activation persists across commands.

4. **Backend entrypoints**: All backend Python entrypoints (scripts, servers, CLIs) MUST be runnable via `.venv/bin/python <script>` or Make targets that reference `.venv`.

### Make Targets Rule
All backend entrypoints MUST have Make targets with:
- **Verbosity**: Each target prints what it's about to do, the command it runs, and the result
- **Documentation**: Each target is documented in README.md with: purpose, prerequisites, usage, what it does, and expected output
- **Idempotency**: Running the target twice should not fail or corrupt state

### Inspection Discipline
- Each ✎ (mutate) command must have at least one ⏾ (inspect) command in its `depends_on` chain that reads the target file BEFORE modification
- ⏾ commands are read-only; ✎ commands modify/create; ✓ commands verify
- Order within a stage: ⏾ → ✎ → ✓ (but re-inspection after mutation is allowed)

### Verification Gate
- No stage proceeds until its local checks pass
- Global checks run only at stage completion
- Never weaken assertions to pass — fix the implementation or test

## Spec → Plan Mapping

1. Read the spec file(s) provided
2. Identify the feature's scope, acceptance criteria, and constraints
3. Break into stages (each independently verifiable)
4. Within each stage, break into atomic commands with explicit `depends_on` chains
5. Add preconditions, verification gates, and failure procedures
6. Include the Python venv precondition in EVERY plan

---

## Command Types

| Marker | Type | Purpose | Example |
|--------|------|---------|---------|
| ⏾ | inspect | Read-only understanding | `read_file`, `search_files`, `ls`, `--version` |
| ✎ | modify/create | Make changes | `write_file`, `patch`, `terminal` (mutating) |
| ✓ | verify | Confirm correctness | `test`, `build`, `lint`, `curl` (health check) |

---

## Output Location

Unless specified otherwise, output to:
- `PLAN.md` — in the directory provided
- `RUNBOOK.md` — in the same directory

Both files should be self-contained — a coding agent with no session context should be able to execute the runbook by following the commands in order.
